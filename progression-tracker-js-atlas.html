Fast Orientation

The shortest useful mental model is this: PHP renders the page shell, the browser downloads a page script, that script listens for clicks and form submits, then it calls a PHP API route with fetch(). The JavaScript layer is small, but it is the layer that makes the app feel interactive instead of full-page-reload-only.

Anchor File

public/assets/js/app.js

const response = await fetch(endpoint, {
    credentials: 'include',
    ...options,
    headers: {
        Accept: 'application/json',
        ...options.headers,
    },
});

This is the shared transport layer. Most dashboard pages use this helper so they can talk to PHP endpoints in one consistent way.

Anchor File

public/assets/js/auth.js

const formRules = {
    loginForm: {
        email: (value) => validateEmail(value),
        password: (value) => validateRequired(value, 'Password'),
    },

This file acts like a front-desk checker. It catches obvious mistakes in the browser before the request ever reaches PHP.

Anchor File

public/assets/js/history.js

const params = new URLSearchParams({
    mode: 'day',
    scope: getPanelScope(panel),
    date,
});

This is the analytics engine in the browser. It builds the right query string, asks the history API for data, and redraws the heatmap, charts, and day modal.

Browser Flow In 3 Moves

Almost every JavaScript action in this app fits the same pattern: boot on page load, react to a user gesture, then call a PHP endpoint and update the visible UI.

Move 1
Boot

A page loads its script with defer. That script waits for DOMContentLoaded so the HTML exists before it starts wiring buttons and forms.

PHP view→script tag→DOMContentLoaded
Move 2
React

The script listens for clicks, key presses, and form submits. It reads values from inputs or data-* attributes already printed by PHP.

button click→read form/card data→validate
Move 3
Call + Refresh

The script sends JSON to a PHP API route. If that succeeds, it either reloads the page or redraws a panel such as the history workspace.

fetch()→public/api/*.php→reload or repaint

What Loads Where

The auth pages load one shared script, while each dashboard page loads app.js, history.js, and a page-specific script. That makes the client layer modular without becoming a complex frontend framework.

Auth Pages

login.php, register.php, and forgot.php all load assets/js/auth.js.

Dashboard Pages

dashboard-start.php always loads app.js and history.js. Individual views then add dashboard.js, categories.js, dreams.js, goals.js, or logs.js.
