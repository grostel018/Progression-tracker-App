What The App Does

From Login Screen To Personal Progress Control Room

This app is a personal tracking system. You log in, store categories, connect dreams to those categories, break dreams into goals, and then turn goals into daily proof through tasks, habits, and history entries.

<?php

declare(strict_types=1);

define('BASE_PATH', dirname(__DIR__));
define('SRC_PATH', __DIR__);

spl_autoload_register(static function (string $class): void {
    $prefix = 'src\\';

    if (strncmp($class, $prefix, strlen($prefix)) !== 0) {
        return;
    }

    $relativeClass = substr($class, strlen($prefix));
    $path = SRC_PATH . '/' . str_replace('\\', '/', $relativeClass) . '.php';

    if (is_file($path)) {
        require_once $path;
    }
});
Plain English

This is the autoloader inside `src/bootstrap.php`. It teaches PHP how to find classes automatically when the code references something like `src\api\goals\GoalController`.

Without this, every file would need a pile of manual `require_once` lines. With it, PHP turns class names into file paths behind the scenes.

That is a strong sign this project wants organized modules, not one giant script.

The View And The Repository Are Deliberately Kept Apart

The dashboard view knows how to display stats cards. The dashboard repository knows how to count dreams, active goals, streaks, and achievements. That split is the whole trick.

Healthy split

The view prints labels and numbers. The repository runs SQL and returns plain data. The page file glues them together.

Bad split

If the view queried MySQL directly, design changes would risk breaking data logic. If the repository returned HTML, backend code would become harder to reuse.

Code 
public function getStats(int $userId): array
{
    $stats = [
        'total_dreams' => 0,
        'active_goals' => 0,
        'current_streak' => 0,
        'achievements' => 0,
    ];

    $stmt = $this->db->prepare('SELECT COUNT(*) FROM dreams WHERE user_id = ?');
    $stmt->execute([$userId]);
    $stats['total_dreams'] = (int) $stmt->fetchColumn();

    $stmt = $this->db->prepare('SELECT COUNT(*) FROM goals WHERE user_id = ? AND status = ?');
    $stmt->execute([$userId, 'active']);
    $stats['active_goals'] = (int) $stmt->fetchColumn();
Plain English

`DashboardRepository` starts with safe zero values so the page has predictable output even if later queries fail or optional tables are missing.

Then it asks the database focused counting questions, one by one: how many dreams does this user have, and how many active goals?

This is classic summary-query code. It is boring in the best possible way, which makes it reliable.

⚠
Common AI Mistake

AI tools often want to “modernize” this into a single-page app because they see some JavaScript. That would be a rewrite, not a refactor. The project’s real architecture is server-rendered first.

Code 
async function apiRequest(endpoint, options = {}) {
    try {
        const response = await fetch(endpoint, {
            credentials: 'include',
            ...options,
            headers: {
                Accept: 'application/json',
                ...options.headers,
            },
        });

        if (response.status === 401) {
            window.location.href = 'login.php';
            return null;
        }

        return await response.json();
    } catch (error) {
        console.error(error);
        return {
            success: false,
            message: 'Request failed. Check your connection and try again.',
        };
    }
}
Plain English

This helper wraps `fetch()` so every request behaves the same way. It includes cookies, asks for JSON, and catches network failures.

If the server answers with `401`, which means “you are not authorized,” the browser is pushed back to `login.php`.

That one function quietly removes a lot of repeated code from the page modules.

Resource Controllers Keep CRUD Predictable

For categories, dreams, and goals, the app uses the same CRUD pattern over and over: GET for list/show, POST for create, PUT for update, DELETE for removal.

G

P

POST

Create new rows such as goals, dreams, categories, tasks, or history entries.

U

PUT

Update existing rows, such as changing goal status or toggling task completion.

D

Remove rows after confirmation, then refresh the UI.

Code 
function dispatch_resource_controller(object $controller): void
{
    $id = request_query_int('id');
    $input = request_input();

    switch (request_method()) {
        case 'GET':
            json_response($id > 0 ? $controller->show($id) : $controller->index());
            return;

        case 'POST':
            json_response($controller->store($input));
            return;

        case 'PUT':
            json_response($controller->update($id, $input));
            return;

        case 'DELETE':
            json_response($controller->delete($id));
            return;
    }
}
Plain English

This helper is a traffic cop for standard JSON endpoints. It inspects the HTTP method and sends the request to the matching controller method.

That means category, dream, and goal APIs can stay tiny because the dispatch pattern is shared.

If you ever add a new CRUD-style feature, this helper tells you what shape the endpoint will probably follow.

public function update(int $taskId, array $input): array
{
    $this->auth->requireAuth();

    try {
        if (array_key_exists('is_completed', $input)) {
            return $this->service->toggleTaskCompletion(
                $this->auth->id(),
                $taskId,
                (bool) $input['is_completed'],
                $input['completed_on'] ?? null
            );
        }

        return $this->service->updateTask($this->auth->id(), $taskId, $input);
    } catch (InvalidArgumentException $exception) {
        return ['success' => false, 'message' => $exception->getMessage()];
    }
}
Plain English

`GoalTaskController` is doing a subtle but useful trick here. The same endpoint can handle two different intentions: editing task text or toggling completion.

It decides which path to use by checking whether the request includes `is_completed`.

That keeps the API surface small while still supporting richer behavior.

Auth And Recovery

Sessions, Security Questions, And The Safety Rails Around The App

This module uses a nightclub metaphor. Authentication is the bouncer, the session is the wristband, and the recovery flow is a second ID check when the guest forgets the original pass.

The Browser Does Some Validation, But The Server Makes The Final Decision

`auth.js` gives fast feedback so the user is not guessing. But the backend still re-validates because browser checks are convenience, not security.

Frontend Role

`auth.js`

1
Trim input

Removes accidental whitespace from emails and usernames.

2
Show field errors

Marks specific inputs with messages like “passwords do not match.”

3
Submit async

Sends the form to JSON endpoints without a full form-post refresh.

Backend Role

Handlers + `Auth.php`

1
Validate again

Never trust the browser as the only gatekeeper.

2
Hash and verify passwords

Passwords are compared using secure hashing, not plain text.

3
Create session state

After successful login, the server stores `user_id` in the session.

Code 
public function handle(array $input): array
{
    $username = trim($input['username'] ?? '');
    $email = trim($input['email'] ?? '');
    $password = $input['password'] ?? '';
    $passwordConfirm = $input['password2'] ?? '';
    $securityQuestions = $input['security_questions'] ?? '';

    $validation = $this->validator->validateRegister($input);
    if ($validation !== []) {
        return ['success' => false] + $validation;
    }

    $result = $this->auth->register($username, $email, $password);
Plain English

`RegisterHandler` pulls the important fields out of the request, normalizes them, and runs server-side validation before it even tries to create a user.

That means the browser’s nice messages are not the only protection. The server repeats the important rules.

This is a core security habit: user input gets checked again at the actual trust boundary.

Password Recovery Is A Three-Step Tunnel

The forgot-password flow is more sophisticated than the login page. It first confirms the email, then loads the user’s saved security questions, then issues a temporary reset token stored in session state.

Step 1

Email lookup

The backend verifies the account exists and has enough recovery questions configured.

Step 2

Answer verification

`SecurityQuestionHandler` checks two answers and then creates a temporary reset token in session memory.

Step 3

Password update

`ResetPasswordHandler` validates the token, checks expiration, hashes the new password, and clears the token.

Code

Plain English

After the answers are verified, the server creates a random reset token and stores it in the session along with the target user and expiry time.

That token acts like a temporary backstage pass. It is only valid for a short window and only for this recovery session.

You should notice one practical limitation too: the comment says this is session-based “for now,” which means a future production version might move reset tokens into the database.
